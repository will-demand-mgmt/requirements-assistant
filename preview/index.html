How it works
You describe. The assistant structures.

No blank page, no template to fill. Tell the assistant what you're building — it asks the right questions and produces a complete, structured document.

I need to create a BRD for a new API that lets customers dynamically reallocate satellite capacity across beams.
Great starting point. To build the Business Objectives table, I need to understand the key drivers. What is the main operational problem this API is solving — over-provisioning, underutilisation, or lack of real-time flexibility?
All three — customers reserve capacity across all beams but terminals only use one beam at a time. We want to enable real-time reallocation with a 15-minute SLA.
Business Objectives (BO table — draft) BO-01  |  Enable dynamic MHz reallocation across beams  |  Eliminate static over-provisioning
BO-02  |  Remove manual intervention from allocation process  |  Reduce operational cost
BO-03  |  Achieve end-to-end execution within 15 minutes  |  Support near real-time operations
Shall I confirm these and move to Scope Definition?
Document modes
Four modes. One consistent
output standard.

Every mode produces structured, table-based documents with unique IDs — ready to be shared with engineering teams or fed into AI demand management systems.

📋
BRD — Business Requirements
Guided elicitation of business context, objectives, stakeholders, RACI matrix, business requirements, constraints, assumptions, glossary, and success criteria.
Output: BO-xx · BR-xx · BC-xx · SC-xx tables
⚙️
FRD — Functional Requirements
Complete functional specification including operational flow, functional requirements, state model with transitions, API specification, data entities, architecture, NFRs, error handling, and traceability matrix.
Output: FR-xx · NFR-xx · state model · API spec
🗂️
User Stories + Acceptance Criteria
Agile-format stories grouped by Epic, with Gherkin acceptance criteria (Given/When/Then), story points on Fibonacci scale, and dependency mapping.
Output: US-xxx · Epic · Given/When/Then · Points
🔍
Review — Document Analysis
Paste an existing requirements document and get a structured quality review: completeness score, [CRITICAL] issues, [IMPROVEMENT] suggestions, and an improved version of the weakest sections.
Output: Score · [CRITICAL] · [IMPROVEMENT] · rewrite
